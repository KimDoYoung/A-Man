# 프론트엔드(Client) 캐싱 및 렌더링 최적화 계획서 (client_caching_and_rendering_plan.md)

## 1. 목표 및 배경 (Goal Description)
백엔드 측의 HTML 변환 캐싱(`@Cacheable`)에 발맞춰, 프론트엔드(클라이언트) 단에서도 불필요한 네트워크 호출과 무거운 마크다운 구문 분석(AST Parsing) 오버헤드를 제어하여 성능을 극대화하고자 합니다.

현재 A-Man 프론트엔드는 사용자가 도움말 트리 메뉴를 이동할 때마다 항상 동일 문서를 서버에서 새로 페치하고 있으며, 폰트 조절 등의 상태 변화 시 마크다운 파서가 무조건 재가동되는 아키텍처적 아쉬움이 있습니다.

---

## 2. 프론트엔드 개선 포인트 3선

### 1) TanStack Query (React Query) 도입을 통한 API 네트워크 캐싱
* **현상**: `MarkdownViewer.tsx`는 단순 `useEffect` 내에서 `apiClient.get`을 직접 찔러 매번 데이터를 가져오고 로딩 스케줄러(Spinner)를 출력합니다.
* **해결책**: `react-query`를 연동하여 조회된 데이터를 고유 쿼리 키(`['manual', pageId]`)에 캐싱합니다.
* **효과**: 사용자가 다른 메뉴를 클릭했다가 직전에 읽던 도움말 화면으로 복귀할 때, **네트워크 지연 없이 0초 만에 기존 화면이 즉시 출력**됩니다.

### 2) `React.useMemo`를 이용한 마크다운 파싱 결과 메모제이션
* **현상**: 도움말 본문 화면에서 폰트 크기 변경(`fontSize`), 넓이 변경(`contentWidth`), 다크 모드 토글 등이 작동할 때 리액트 컴포넌트가 리렌더링됩니다. 이때 본문 텍스트(`page.content`)가 100% 동일함에도 불구하고, 수십KB에 달하는 문자열을 파싱하는 `<ReactMarkdown>` 엔진이 처음부터 AST 구문 분석을 매번 다시 실행하여 브라우저 연산 렉을 유발합니다.
* **해결책**: 파싱 결과를 `useMemo`로 묶어 본문 문자열이 변경될 때만 파싱 연산이 구동되도록 제어합니다.
* **효과**: 폰트 크기 변환이나 목차(TOC) 클릭 시 리액트 가상 DOM 연산이 대폭 감소해 화면 프레임 드랍이 방지됩니다.

### 3) 에디터 타이핑 입력 시 실시간 미리보기 디바운싱(Debouncing) 적용
* **현상**: 관리자가 도움말을 수정할 때, 키보드를 한 자 입력할 때마다 실시간으로 수십KB 문서를 HTML로 변환하여 실시간 뷰어를 갈아끼웁니다. 타이핑 속도가 렌더링 변환 속도보다 빠르면 타자 밀림 렉이 심하게 일어납니다.
* **해결책**: 사용자가 타자 입력을 시작하면 변환을 홀딩하다가, 입력을 멈추고 `150ms ~ 200ms`가 흐르면 최종 마크다운 소스를 미리보기 창으로 일괄 흘려 렌더링을 1회만 타도록 디바운싱 필터를 적용합니다.
* **효과**: 입력감이 100% 부드러워지며 불필요한 연속 렌더링 횟수를 90% 이상 절감할 수 있습니다.

## 3. 잠재적 부작용 및 방어 전략 (Side Effects & Mitigations)

클라이언트 단 최적화 과정에서 발생할 수 있는 부작용과 이를 원천 차단하기 위한 설계 기법입니다.

### 부작용 A: 문서 수정 후 이전 화면 조회 시 옛날 본문 노출 (Stale Data)
* **설명**: API 조회 결과를 캐싱할 때, 관리자가 문서를 수정하고 저장한 직후 뷰어 화면으로 진입하면 브라우저 메모리에 남아 있는 이전 캐시본(`staleTime` 이내)이 화면에 보임으로써 수정 사항이 즉시 갱신되지 않는 현상입니다.
* **방어 전략**:
  * 관리자가 수정을 완료(Save/Update)하는 Mutation API 성공 핸들러 시점에 `queryClient.invalidateQueries({ queryKey: ['manual', pageId] })`를 강제 실행합니다.
  * 해당 키의 캐시를 **즉시 만료(Invalidate)**시킴으로써, 다음 뷰어 진입 시 최신 마크다운을 백엔드에서 강제로 강제 호출하도록 보장합니다.

### 부작용 B: useMemo 캐싱 후 폰트 조절/다크모드 등이 무시되는 현상
* **설명**: `useMemo`의 의존성 배열(`deps`)에 본문 텍스트(`page.content`)만 명시할 경우, 사용자가 폰트 크기를 변경하거나 다크모드를 켜도 파싱 결과가 메모리 캐시본을 재사용하게 되면서 **변경된 스타일이 화면에 반영되지 않는 현상**입니다.
* **방어 전략**:
  * 렌더링 스타일에 직접 기여하는 외부 상태값(`fontSizeClassMap`, 다크모드 상태, `settings`)을 `useMemo` 의존성 배열에 반드시 함께 바인딩합니다.
  * `}, [page?.content, fontSizeClassMap, settings])`

### 부작용 C: 타이핑 후 즉시 저장 시 마지막 글자 누락 (Race Condition)
* **설명**: 에디터에 타이핑 중 마지막 한두 자를 입력하고 디바운스 딜레이(`200ms`)가 지나기 전에 '저장'을 누르면, 디바운스되지 않은 예전 텍스트가 서버로 전송되어 수정본 끝자리가 유실되는 현상입니다.
* **방어 전략**:
  * 디바운싱은 오직 **화면 출력용 렌더링 대상(Preview Text)**에만 비동기로 적용합니다.
  * 실제 에디터가 소유한 폼 원본 원천 값은 디바운스 없이 실시간으로 React State나 Form State에 저장되며, 저장 요청 시에는 이 **원본 텍스트**를 전송하게 설계하여 유실을 완벽히 방지합니다.

## 4. 검토 피드백 요청 (Open Questions)

> [!IMPORTANT]
> 1. 현재 프로젝트가 **React 19.x 및 Zustand** 환경을 갖추고 있습니다. TanStack Query(`@tanstack/react-query`) 라이브러리는 이미 프로젝트 의존성에 등재되어 있는 상태인지 검토하고, 없는 경우 추가가 필요할 수 있습니다.
> 2. 위 개선 방안 중 백엔드 캐싱(`@Cacheable`) 구현 시 프론트엔드의 **2) useMemo 메모제이션** 및 **3) 에디터 디바운싱**을 우선 패키지로 엮어서 최적화 작업을 함께 기획하는 것이 효율적일지 의견을 구합니다.
