## jskn 서버에 배포

### 상황

1. jskn은 docker에 여러 app을 올려 놓고 서비스한다.
2. nginx로 reverse proxy를 사용하고 있다.
3. jskn서버의 docs/docker-compose.yml과 default.conf 을 확인하기 바람.
4. A-Man app은 자바 1.8 임으로 jskn의 tomcat을 사용할 수 없다.
5. A-Man의 자체 image를 만들어서 따로 docker에 올려야 한다.
6. 기존 aview 서비스가 자바1.8로 만들어져서 비슷한 상황이다.
7. jskn 파일 위치
    /data/docker/docker-compose.yml
    /data/docker/nginx/conf.d/default.conf
8. 내가 jskn서버에 /data/docker/apps/aman/db 을 생성하고 aman.db를 생성해 둠

### 방법

1. jskn 서버 : /data/docker/apps/aman/data 폴더를 생성해 둠 
2. Dockerfile을 생성
3. deploy-jskn.sh 에서 aman.war를 만듬
4. deploy-jskn.sh 에서 /data/docker/apps/aman/data으로  아래 2개의 파일 업로드
    1. aman.war 
    2. Dockerfile 
5. jskn 서버의 docker-compose.yml 
```yml
  aman:
    build:
      context: /data/docker/apps/aman/data   # Dockerfile과 aview.war가 있는 경로
      dockerfile: Dockerfile
    container_name: jskn_aman
    user: "1000:1000" # kdy987
    restart: always
    # Nginx가 내부망(8080)으로 통신하므로 외부 포트는 열지 않아도 되지만, 
    # 초기 테스트용으로 8081 등을 열어둘 수 있습니다.
    ports:
      - "8686:8080"
    volumes:
      - /data/docker/apps/aman:/data/docker/apps/aman:Z
    environment:
      - TZ=Asia/Seoul
      - AMAN_MODE=jskn
      - SPRING_PROFILES_ACTIVE=jskn
```
6. jskn서버의 default.conf 수정

```
    location /aman {
        proxy_pass http://aman:8080/aman; # 도커 네트워크 브릿지를
  통해 컨테이너 명으로 통신
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
```
7. jskn  docker 재기동
8. nginx 재기동

        # --- aman (AssetERP 매뉴얼 시스템) 설정 추가 ---
        location /aman/ {
            proxy_pass http://jskn_aman:8080/aman/;

            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;

            # 대용량 파일 및 마크다운 업로드를 위해 제한 용량 및 타임아웃 상향
            client_max_body_size 50M;
            proxy_read_timeout 120s;
        }

## 배포 후 문제점 및 조치 사항

배포 진행 과정 및 배포 직후 발생했던 주요 문제점들과 이에 대한 조치 내역입니다.

### 1. Tomcat 권한 오류 (`Unable to create directory /usr/local/tomcat/webapps/aman`)
* **원인**: Tomcat 컨테이너가 `1000:1000` 비루트 계정으로 실행되면서 Tomcat webapps 및 데이터 디렉토리에 생성/쓰기 권한이 없어 WAR 압축 해제에 실패함.
* **조치**: [Dockerfile](file:///home/kdy987/work/aman/Dockerfile) 내에 `RUN chown -R 1000:1000 /usr/local/tomcat /data/docker/apps/aman` 구문을 추가하여 디렉토리 소유권을 부여함.

### 2. Spring Boot 외부 Tomcat 배포 시 404 및 보안 접근 거부
* **원인**: 외부 Tomcat 배포를 위한 `SpringBootServletInitializer` 상속 누락 및 `SecurityConfig` 서블릿 경로 보안 권한 설정 오류.
* **조치**: [AmanApplication.java](file:///home/kdy987/work/aman/backend/src/main/java/kr/co/kfs/aman/AmanApplication.java)에 `SpringBootServletInitializer` 상속 및 `configure()` 재정의를 적용하고, [SecurityConfig.java](file:///home/kdy987/work/aman/backend/src/main/java/kr/co/kfs/aman/config/SecurityConfig.java)의 권한 설정을 `.antMatchers("/admin/**").authenticated().anyRequest().permitAll()`로 업데이트함.

### 3. Nginx 헤더 494 Error & 정적 자원 라우팅 충돌(400 Bad Request)
* **원인**: Nginx 기본 클라이언트 헤더 버퍼 크기 부족(494) 및 Vite 기본 빌드 자원 폴더(`assets`)가 백엔드 REST Controller API(`@RequestMapping("/assets")`) 경로와 충돌하여 400 에러 발생.
* **조치**: Nginx 설정([docs/default.conf](file:///home/kdy987/work/aman/docs/default.conf))에 `client_header_buffer_size 8k;`, `large_client_header_buffers 4 32k;` 추가 및 [frontend/vite.config.ts](file:///home/kdy987/work/aman/frontend/vite.config.ts)의 `assetsDir`을 `'static-assets'`로 변경함.

### 4. SPA 서브경로 라우팅 누락 (`/aman` 누락 및 새로고침 시 404 Whitelabel Error)
* **원인**: React Router `basename` 미설정으로 인한 주소창 `/aman` 누락 문제 및 브라우저 직접 접속/새로고침(F5) 시 Spring Boot 백엔드 컨트롤러 미매핑으로 인한 404 Whitelabel Error 발생.
* **조치**: 
  - [frontend/src/App.tsx](file:///home/kdy987/work/aman/frontend/src/App.tsx)에 `basename: '/aman'` 설정 추가.
  - [SpaErrorViewResolver.java](file:///home/kdy987/work/aman/backend/src/main/java/kr/co/kfs/aman/config/SpaErrorViewResolver.java)를 구현하여 브라우저의 SPA 경로 404 요청을 `index.html`로 포워딩(200 OK) 처리.
  - [DocsController.java](file:///home/kdy987/work/aman/backend/src/main/java/kr/co/kfs/aman/controller/DocsController.java)의 `/{page_id}` 경로 매핑에 정규식(`/{page_id:\\d+}`) 추가.

### 5. HTTP 환경 클립보드 복사 미동작 문제
* **원인**: `navigator.clipboard.writeText` API는 보안 컨텍스트(HTTPS)에서만 동작하여 HTTP 배포 환경에서 URL 복사 버튼 작동 불가.
* **조치**: [DocUserMain.tsx](file:///home/kdy987/work/aman/frontend/src/domains/content/DocUserMain.tsx)에 `navigator.clipboard` 실패 시 호환용 임시 `textarea` 및 `document.execCommand('copy')`를 수행하는 Fallback 함수(`copyTextToClipboard`) 적용.
