대규모 소프트웨어(특히 자산운용사용 ERP처럼 기능이 복잡하고 방대한 시스템)의 매뉴얼 사이트는 사용자가 원하는 정보를 **'1초 안에 찾고 직관적으로 이해할 수 있도록'** 돕는 것이 최우선 목적입니다.

이를 위해 전 세계적인 표준 매뉴얼 UI(Stripe Docs, 아틀라시안 Confluence, GitBook 등)는 대개 다음과 같은 **'3단 분할 레이아웃(3-Column Layout)'** 구조를 기본 뼈대로 채택합니다.

---

## 🏛️ 표준 매뉴얼 사이트의 핵심 UI 구조

### 1. 좌측 영역 (Left Sidebar): 계층형 내비게이션 트리 (Navigation Tree)

방대한 ERP 메뉴 구조를 한눈에 파악할 수 있도록 돕는 영역입니다.

* **트리 구조(Tree View):** `자산운용` $\rightarrow$ `펀드 설정` $\rightarrow$ `신규 펀드 등록` 처럼 뎁스(Depth)가 있는 아코디언 메뉴 방식을 사용합니다.
* **상태 표시:** 사용자가 현재 읽고 있는 메뉴가 어디인지 하이라이트 처리가 되어야 합니다.
* **구현 컴포넌트:** `shadcn/ui`의 Accordion이나 Sidebar 컴포넌트 조합으로 깔끔하게 구현 가능합니다.

### 2. 중앙 영역 (Main Content): 마크다운 기반의 본문

실제 매뉴얼 내용이 표시되는 핵심 영역입니다. `AMan` 프로젝트의 경우 Milkdown 에디터로 작성된 마크다운이 HTML로 렌더링되어 보여지는 곳입니다.

* **타이틀 및 작성 정보:** 상단에 문서 제목, 최종 수정일, 수정자 등이 명시됩니다.
* **콘텐츠 가독성:** 폰트 크기, 줄 간격, 그리고 특히 **표(Table)** 스타일링이 중요합니다. ERP 매뉴얼 특성상 입력 필드 설명이나 데이터 그리드 항목 설명용 표가 많이 들어가기 때문입니다.
* **콜아웃(Callout) 박스:** 주의사항(`Warning`), 팁(`Tip`), 중요 안내(`Note`) 등을 시각적으로 분리하여 가독성을 높입니다.

### 3. 우측 영역 (Right Sidebar): 문서 내 목차 (Table of Contents - TOC)

현재 읽고 있는 '하나의 문서' 안에서 빠르게 이동할 수 있도록 돕는 네비게이터입니다.

* **스크롤 스파이(ScrollSpy):** 사용자가 본문을 아래로 스크롤하면, 본문의 제목(`H2`, `H3`)을 감지하여 우측 목차의 해당 항목이 자동으로 활성화됩니다.
* **앵커 링크:** 목차를 클릭하면 본문의 해당 위치로 부드럽게 스크롤(Smooth Scroll)됩니다.

## 🔍 대규모 매뉴얼의 승부처: '글로벌 상단 바 (Header)'

매뉴얼 사이트의 UX를 결정짓는 핵심은 상단 바에 위치한 **'검색(Search)'** 기능입니다.

* **커맨드 센터형 검색창:** `Ctrl + K` (또는 `Cmd + K`) 단축키를 누르면 화면 중앙에 다이얼로그가 뜨면서 전체 문서를 검색할 수 있는 UI가 대세입니다.
* **실시간 자동 완성:** 키워드를 입력하면 문서 제목뿐만 아니라 본문 내용까지 실시간으로 매칭하여 리스트를 보여줍니다.
* **구현 컴포넌트:** `shadcn/ui`의 `Command` 컴포넌트를 활용하면 아주 손쉽게 고급스러운 검색 UI를 연동할 수 있습니다.

## 💡 `AMan` 프로젝트에 제안하는 UI 특화 포인트

자산운용사 사용자를 타겟으로 하는 `AMan`은 일반 오픈소스 문서 사이트와 달리 아래 두 가지 UI가 추가되면 만족도가 극대화됩니다.

1. **AG Grid 기반의 데이터 사전 매핑 UI:** ERP 특성상 화면에 표가 엄청나게 많습니다. 매뉴얼 본문에 정적인 표만 넣지 말고, 프론트 스택에 포함된 `AG Grid`를 활용해 "이 화면의 컬럼 속성 정의서"를 소형 그리드로 제공하면 사용자가 필터링·정렬해가며 매뉴얼을 사전처럼 조회할 수 있습니다.
2. **역할 기반(Role) 메뉴 필터링:** 운용역, 컴플라이언스(준법), 백오피스(회계) 등 로그인한 관리자 권한(`Spring Security`와 연동)에 따라 좌측 내비게이션 트리에서 본인 업무와 연관된 매뉴얼 카테고리만 먼저 접혀있거나 활성화되도록 제어하면 복잡도를 크게 낮출 수 있습니다.
