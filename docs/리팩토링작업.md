# 리팩토링 작업 기록

## 2026-07-21: ActionImageEditor.tsx Safe 범위 리팩토링

### 배경
`frontend/src/components/imageditor/ActionImageEditor.tsx`가 4,156줄짜리 단일 컴포넌트로 비대해져 구조 검토 요청. 탐색 결과 핵심 문제는 아이템 타입(circle-number/box/arrow/orthogonal-arrow/text/symbol/image/callout/block-arrow-stamp)별 분기 로직이 draw(), hit-test, resize, crop, margin 계산 등 8~10곳에서 각각 독립적으로 재구현되어 있었다는 점. 순수 헬퍼 함수(거리 계산, 화살표 그리기, 파일 로딩 등)도 컴포넌트 파일 안에 섞여 있었음.

테스트 코드가 전무하고(`*.test.*`/`*.spec.*` 없음, `package.json`에 `test` 스크립트 없음) 개발 서버 직접 실행도 금지되어 있어(`CLAUDE.md` 5항) 런타임 회귀를 잡을 안전망이 `tsc`/`lint`/수동 diff 리뷰뿐이었음. 이 때문에 위험도가 낮고 되돌리기 쉬운 "Safe 범위"만 처리하기로 사용자와 협의 (계획 파일: `~/.claude/plans/frontend-src-components-imageditor-acti-expressive-candle.md`).

### 처리한 작업

**Stage A — 순수 모듈 레벨 헬퍼 분리**
- `canvasDrawHelpers.ts` (신규, 295줄): `getDistanceToSegment`, `getTextBounds`, `drawArrow`, `drawOrthogonalArrow`, `drawSelectionHandles`, `createRoundedRectPath`
- `fileHelpers.ts` (신규, 54줄): `copyTextToClipboard`, `readFileAsDataURL`, `loadImageElement`, `loadImageFromFile`
- 로직 변경 없는 순수 이동

**Stage B1 — 아이템 타입별 geometry(bounds/translate) 통합**
- `itemGeometry.ts` (신규, 126줄)
- `getItemEdgeBounds(item, defaults)`: `minBottomMargin`/`minRightMargin`/`canShrinkTop`/`canShrinkLeft` 4개 useMemo에서 축만 바꿔 중복 구현되던 per-type 분기를 하나로 통합 (실제 로직 통합, 수식 검증 완료)
- `shiftItemX`/`shiftItemY`: 그대로 이관 (병합 안 함 — 각 축이 독립적으로 다른 조건(midX vs midY)을 체크해 병합 시 미묘한 동작 차이 위험)
- `offsetItemForCrop`/`isItemWithinCropBounds`: `cropImage`의 두 체인을 그대로 이관 (조건 구조가 shiftItemX/Y와 달라 병합하지 않고 순수 이동만)
- draw() 렌더 루프, handleMouseDown/handleMouseMove의 hit-test·resize 로직은 canvas context 부작용·이벤트 좌표 의존이 얽혀 있어 이번 범위에서 의도적으로 제외 (B2로 후속 작업 남김)

**Stage C1 — `handleDeleteUserSettings` 버그 수정**
- 기존: ~40개 스타일 필드를 개별 setter로 리셋 (같은 목적의 `handleResetToEmpty`는 `applyStyleConfig(SYSTEM_ITEM_DEFAULTS)` 한 번 호출로 처리 — 복사-붙여넣기 불일치)
- 수정: `applyStyleConfig(SYSTEM_ITEM_DEFAULTS)` 호출로 교체 + `arrowHeadSize`는 `StyleConfig`에 없어 `applyStyleConfig`가 리셋 안 하므로 별도 `setArrowHeadSize` 유지 (`handleResetToEmpty`와 동일 패턴)
- 부수 효과: `textRotation`도 함께 리셋됨 (기존엔 안 건드렸으나 `handleResetToEmpty`와 동작 통일하는 의도된 변경)

### 결과
- `ActionImageEditor.tsx`: 4,156줄 → 3,640줄 (약 516줄 감소)
- 검증: `tsc --noEmit`, `eslint . --ext ts,tsx --max-warnings 0`, `npm run build` 전부 통과
- 커밋은 하지 않음 (작업 트리에 남아있는 변경 상태)

### 이번 범위에서 제외 (후속 작업 후보)
- **Stage B2**: hit-test/resize-handle을 `itemGeometry.ts`로 통합 + `handleMouseMove`의 arrow/orthogonal-arrow 중복 분기 정리
- draw() 렌더 루프/프리뷰 분기 통합
- **Stage C2**: ~50개 useState → reducer/config 객체 통합 + `FloatingPropertyPanel.tsx`의 ~70개 prop 계약 재설계 (가장 큰 ripple, 별도 계획 필요)
- **Stage D**: image-work CRUD를 `useRecentPagesStore` 패턴을 따르는 zustand store로 이전
- 커스텀 React 훅(useUndoRedo, useCanvasDrawing 등) 도입은 이 코드베이스에 선례 없는 패턴이라 비권장 (zustand store가 기존 컨벤션)

## 2026-07-21: 액션바(푸터) JSX → ImageEditorFooter.tsx 분리

### 배경
Safe 범위 리팩토링 이후에도 `ActionImageEditor.tsx`가 여전히 3,640줄. 사용자가 하단 "액션바 (푸터) 영역" `<div>`를 별도 컴포넌트로 떼어내자고 요청. 사전 검토 결과: 위험도 낮은 손쉬운 정리이지만 진짜 문제(타입별 분기 중복)는 안 건드리는 모듈성 개선 정도의 가치 — 사용자에게 이 트레이드오프 안내 후 진행 승인받음.

### 처리한 작업
- `ImageEditorFooter.tsx` (신규, 130줄): 푸터 전체 JSX를 프레젠테이션 컴포넌트로 분리. `WorkHistory.tsx`의 "데이터 프리미티브 + `on*` 콜백" prop 컨벤션을 그대로 따름. props 13개: `bgImage`, `savingWork`, `isDirty`, `insertingImage`, `generatedImageUrl` + 8개 콜백.
- `handleCopyGeneratedUrl`/`handleCopyMarkdownUrl` 신설 (`ActionImageEditor.tsx`, `handleGenerateUrl` 바로 뒤): "주소 복사"/"마크다운 복사" 버튼에 인라인으로 박혀 있던 try/catch 2개를 named 함수로 승격 — 다른 버튼들과 동일하게 "콜백 하나 = named handler 하나" 패턴으로 통일. 로직 변경 없음.
- `ActionImageEditor.tsx`의 lucide-react import에서 `Download`/`Copy`/`Save` 제거 (푸터 전용, 새 파일로 이동). `Check`는 다른 곳(확장방향 라디오 버튼)에서도 쓰여 유지.
- 업로드 버튼은 `onUploadClick` 콜백으로 받아 `fileInputRef`는 부모에만 남김 — 실제 `<input type="file">` 엘리먼트도 그대로 부모(메인 캔버스 영역)에 위치.

### 결과
- `ActionImageEditor.tsx`: 3,640줄 → 3,571줄
- 검증: `tsc --noEmit`, `eslint . --ext ts,tsx --max-warnings 0`, `npm run build` 전부 통과
- 커밋 안 함
