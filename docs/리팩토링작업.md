# 리팩토링 작업 기록

## 2026-07-21: ActionImageEditor.tsx Safe 범위 리팩토링

### 배경
`frontend/src/components/imageditor/ActionImageEditor.tsx`가 4,156줄짜리 단일 컴포넌트로 비대해져 구조 검토 요청. 탐색 결과 핵심 문제는 아이템 타입(circle-number/box/arrow/orthogonal-arrow/text/symbol/image/callout/block-arrow-stamp)별 분기 로직이 draw(), hit-test, resize, crop, margin 계산 등 8~10곳에서 각각 독립적으로 재구현되어 있었다는 점. 순수 헬퍼 함수(거리 계산, 화살표 그리기, 파일 로딩 등)도 컴포넌트 파일 안에 섞여 있었음.

테스트 코드가 전무하고(`*.test.*`/`*.spec.*` 없음, `package.json`에 `test` 스크립트 없음) 개발 서버 직접 실행도 금지되어 있어(`CLAUDE.md` 5항) 런타임 회귀를 잡을 안전망이 `tsc`/`lint`/수동 diff 리뷰뿐이었음. 이 때문에 위험도가 낮고 되돌리기 쉬운 "Safe 범위"만 처리하기로 사용자와 협의 (계획 파일: `~/.claude/plans/frontend-src-components-imageditor-acti-expressive-candle.md`).

### 처리한 작업

**Stage A — 순수 모듈 레벨 헬퍼 분리**
- `canvasDrawHelpers.ts` (신규, 295줄): `getDistanceToSegment`, `getTextBounds`, `drawArrow`, `drawOrthogonalArrow`, `drawSelectionHandles`, `createRoundedRectPath`
- `fileHelpers.ts` (신규, 54줄): `copyTextToClipboard`, `readFileAsDataURL`, `loadImageElement`, `loadImageFromFile`
- 로직 변경 없는 순수 이동

**Stage B1 — 아이템 타입별 geometry(bounds/translate) 통합**
- `itemGeometry.ts` (신규, 126줄)
- `getItemEdgeBounds(item, defaults)`: `minBottomMargin`/`minRightMargin`/`canShrinkTop`/`canShrinkLeft` 4개 useMemo에서 축만 바꿔 중복 구현되던 per-type 분기를 하나로 통합 (실제 로직 통합, 수식 검증 완료)
- `shiftItemX`/`shiftItemY`: 그대로 이관 (병합 안 함 — 각 축이 독립적으로 다른 조건(midX vs midY)을 체크해 병합 시 미묘한 동작 차이 위험)
- `offsetItemForCrop`/`isItemWithinCropBounds`: `cropImage`의 두 체인을 그대로 이관 (조건 구조가 shiftItemX/Y와 달라 병합하지 않고 순수 이동만)
- draw() 렌더 루프, handleMouseDown/handleMouseMove의 hit-test·resize 로직은 canvas context 부작용·이벤트 좌표 의존이 얽혀 있어 이번 범위에서 의도적으로 제외 (B2로 후속 작업 남김)

**Stage C1 — `handleDeleteUserSettings` 버그 수정**
- 기존: ~40개 스타일 필드를 개별 setter로 리셋 (같은 목적의 `handleResetToEmpty`는 `applyStyleConfig(SYSTEM_ITEM_DEFAULTS)` 한 번 호출로 처리 — 복사-붙여넣기 불일치)
- 수정: `applyStyleConfig(SYSTEM_ITEM_DEFAULTS)` 호출로 교체 + `arrowHeadSize`는 `StyleConfig`에 없어 `applyStyleConfig`가 리셋 안 하므로 별도 `setArrowHeadSize` 유지 (`handleResetToEmpty`와 동일 패턴)
- 부수 효과: `textRotation`도 함께 리셋됨 (기존엔 안 건드렸으나 `handleResetToEmpty`와 동작 통일하는 의도된 변경)

### 결과
- `ActionImageEditor.tsx`: 4,156줄 → 3,640줄 (약 516줄 감소)
- 검증: `tsc --noEmit`, `eslint . --ext ts,tsx --max-warnings 0`, `npm run build` 전부 통과
- 커밋은 하지 않음 (작업 트리에 남아있는 변경 상태)

### 이번 범위에서 제외 (후속 작업 후보)
- **Stage B2**: hit-test/resize-handle을 `itemGeometry.ts`로 통합 + `handleMouseMove`의 arrow/orthogonal-arrow 중복 분기 정리
- draw() 렌더 루프/프리뷰 분기 통합
- **Stage C2**: ~50개 useState → reducer/config 객체 통합 + `FloatingPropertyPanel.tsx`의 ~70개 prop 계약 재설계 (가장 큰 ripple, 별도 계획 필요)
- **Stage D**: image-work CRUD를 `useRecentPagesStore` 패턴을 따르는 zustand store로 이전
- 커스텀 React 훅(useUndoRedo, useCanvasDrawing 등) 도입은 이 코드베이스에 선례 없는 패턴이라 비권장 (zustand store가 기존 컨벤션)

## 2026-07-21: 숫자키(1~9)로 도구 전환 단축키 추가

### 배경
사용자가 도형/텍스트 도구 바꿀 때마다 마우스로 좌측 툴바까지 왕복하는 게 작업 능률에 안 좋다고 지적. `Esc`가 이미 `pointer`(선택/이동)로 바로 전환해주는 단축키로 동작 중이라, 같은 방식으로 숫자키 1~9에 나머지 도구를 매핑. 사용자가 예시로 든 "1=원숫자, 2=사각형"이 실제 좌측 툴바 등록 순서와 일치해 그 순서 그대로 사용.

### 처리한 작업
- `ActionImageEditor.tsx`의 전역 키보드 단축키 `useEffect` 안, `Escape` 분기 바로 뒤에 숫자키 분기 추가.
- `NUMBER_KEY_TOOLS` 매핑(컴포넌트 내부, `useEffect` 바로 위): `1`=원숫자(`circle-number`), `2`=박스(`box`), `3`=화살표(`arrow`), `4`=꺾임화살표(`orthogonal-arrow`), `5`=스탬프(`block-arrow-stamp`), `6`=말풍선(`callout`), `7`=이모지(`symbol`), `8`=텍스트(`text`), `9`=크롭(`crop`) — 좌측 툴바 등록 순서와 동일.
- 툴바 버튼 클릭과 동일하게 `setActiveTool` + `setSelectedItemId(null)` 둘 다 실행.
- Ctrl/Cmd/Alt 조합 시엔 무시(다른 단축키·브라우저 기본 동작과 충돌 방지). 텍스트 입력 중 가드는 기존 keydown effect의 `document.activeElement` 체크를 그대로 재사용 — 새 가드 추가 안 함.
- 새 파일 없음, 기존 핸들러/가드만 재사용.

### 결과
- 검증: `tsc --noEmit`, `eslint . --ext ts,tsx --max-warnings 0`, `npm run build` 전부 통과
- 실제 키 입력 동작은 개발서버 직접 실행 금지 정책상 사용자가 `fm.sh run`으로 직접 확인 필요
- 커밋 안 함

## 2026-07-21: 페이지 관리 목록을 폴더(메뉴) 트리 순서로 정렬

### 배경
`PageManagePage.tsx`(admin 전용) 목록이 백엔드 `searchPages` 쿼리의 `ORDER BY p.id DESC`(등록 역순)로만 나와, 사용자가 메뉴 폴더 트리 순서에 맞춰 보고 싶다고 요청.

### 확인된 사실 / 발견한 버그성 갭
- `Folder.sortOrder`는 항상 세팅되는 신뢰 가능한 정렬 키. `Folder.nums`("1.2.3" 형식)는 `regenerate-all`을 눌러야만 최신화되는 값이라 신뢰 불가.
- `Folder.java`의 `@OneToMany children`(68~70줄)에 `@OrderBy` 없어서, `/docs/folders` 응답에 중첩되는 `children` 배열의 순서가 `sortOrder` 기준으로 보장되지 않던 상태였음(다른 명시적 조회 메서드 `findByParentIdOrderBySortOrderAsc`들은 정렬 보장되는데 이 관계 필드만 예외).

### 처리한 작업
- **백엔드** `Folder.java`: `children` 필드에 `@OrderBy("sortOrder ASC")` 추가 — nested 직렬화 순서를 항상 sortOrder 기준으로 고정.
- **프론트** `PageManagePage.tsx`:
  - 폴더 트리 조회 시(`flattenAndCache`) DFS 순회 인덱스를 `folderOrderIndex`(Map)로 같이 구축.
  - `sortedRowData`(`useMemo`) 신설: `rowData`를 "폴더 DFS 순서 → 같은 폴더 내 `page.sortOrder`" 2단 기준으로 정렬. 폴더 없는 페이지는 맨 뒤로.
  - `AgGridReact`의 `rowData`를 `sortedRowData`로 교체. 백엔드 `ORDER BY p.id DESC`는 그대로 둠(다른 화면 영향 방지 목적, 정렬은 프론트에서 최종 재정렬).

### 결과
- 검증: 백엔드 `./bm.sh compile` 통과, 프론트 `tsc --noEmit`/`eslint`/`npm run build` 전부 통과
- 실제 화면 확인은 사용자가 `bm.sh run`/`fm.sh run`으로 직접 필요
- 커밋 안 함

## 2026-07-21: 페이지 관리에 1단계/2단계 폴더 필터 드롭다운 추가

### 배경
`PageManagePage.tsx`에 키워드/상태/잠금 필터는 있었지만 메뉴(폴더) 단위 필터가 없어, 1단계·2단계 메뉴를 드롭다운으로 골라 좁혀보고 싶다는 요청. cascading 방식: 1단계 고르면 2단계 옵션이 하위로 좁혀지고, 2단계 먼저 고르면 1단계도 자동 반영. 나머지 필터와 AND, 초기화 시 둘 다 전체로.

### 가능여부/난이도 판단
- 백엔드 변경 불필요로 결론: `GET /admin/page-list`가 페이지네이션 없이 전량 반환(`AdminPageController.java`, `PageRepository.searchPages`), 폴더 트리도 이미 마운트 시 전량(479건: 1단계 13/2단계 87/3단계 379) 클라이언트에 로드돼 있어 순수 클라이언트 필터로 처리 — 네트워크 왕복 없어 조회속도 사실상 즉시.
- `Folder.level`(1/2/3)이 이미 JSON에 포함돼 내려옴 확인, 별도 API 불필요.

### 처리한 작업 (전부 `PageManagePage.tsx`)
- `folderMap` 캐시 항목에 `level` 필드 추가.
- `level1Filter`/`level2Filter` state 신설(`number | 'ALL'`).
- `level1Options`/`level2Options`(`useMemo`): `folderMap`에서 레벨별 필터링 + `folderOrderIndex` 기준 정렬. `level2Options`는 `level1Filter` 선택 시 그 하위로만 좁힘.
- `handleLevel1Change`: 1단계를 '전체'로 바꾸면 2단계도 '전체'로 같이 리셋(사용자 확인 완료). 특정 값으로 바꾸면 기존 2단계가 그 자식이 아닐 때만 2단계 리셋.
- `handleLevel2Change`: 특정 값 선택 시 그 폴더의 `parentId`를 찾아 1단계에 자동 반영. '전체' 선택 시 1단계는 유지.
- `folderFilteredRowData`(`useMemo`): `rowData`에 조상 체인 매칭(폴더 id가 선택된 1단계/2단계를 상위 체인에 포함하는지)으로 AND 필터 적용, 기존 `sortedRowData`가 이 결과를 받아 정렬하도록 연결.
- `handleReset`에 `level1Filter`/`level2Filter` '전체' 리셋 추가.
- 검색 폼에 `<select>` 2개 추가 (기존 상태/잠금 필터와 동일한 스타일).

### 결과
- 검증: `tsc --noEmit`, `eslint . --ext ts,tsx --max-warnings 0`, `npm run build` 전부 통과
- 실제 드롭다운 동작(옵션 좁혀짐, cascading, 리셋)은 사용자가 `fm.sh run`으로 직접 확인 필요
- 커밋 안 함
