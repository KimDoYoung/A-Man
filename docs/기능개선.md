# 사용자 환경조정

> 코딩은 하지 말고. 의견만 줘. 표현되는 contents의 width와 text font 크기를 사용자가 조정하게도 할 수 있는가?

▸ Thought for 6s, 750 tokens
  Considering User Interface Adjustments
  사용자가 본문 콘텐츠의 너비(Width)와 텍스트 글자 크기(Font Size)를 화면에서 직접 조절할 수 있도록 구현하는 것은
  충분히 가능하며, 기술적으로도 간단하게 적용할 수 있습니다.

  구현 방식과 사용자 경험(UX) 측면에서의 검토 의견을 정리해 드립니다.
  ──────
  ### 1. 기술적 구현 방안 (원리)

  • 사용자 설정 저장 및 관리 (Local Storage & Zustand)
      • 사용자가 지정한 너비와 글자 크기 값을 브라우저의  localStorage 에 보관하여, 페이지를 새로고침하거나 나중에
      재방문하더라도 사용자가 설정한 값이 그대로 유지되도록 처리합니다.
      • 프론트엔드 내의 전역 상태 관리 라이브러리(Zustand)를 사용하여 설정 창과 본문 뷰어 컴포넌트 간에 조절된 값을
      실시간으로 동기화합니다.
  • UI/UX 배치 제안
      • 도움말 상세 화면의 우측 상단 툴바 영역(또는 목차 상단)에 작은 조절 도구(글자 크기  A- / A+  버튼, 본문 너비
      좁게 / 보통 / 넓게  토글 또는 슬라이더)를 제공합니다.
  • 동적 스타일 적용
      • 본문 영역인  <article>  태그의 CSS 클래스 또는 인라인 스타일을 사용자의 선택 값에 따라 동적으로 변경합니다.
      • 너비 조절:  max-w-3xl (좁게),  max-w-5xl (보통),  max-w-7xl (넓게) 등 미리 정의된 단계를 클래스로 바인딩하거나
      슬라이더 값을 픽셀( px ) 단위로 인라인 스타일의  maxWidth  값에 직접 매핑합니다.
      • 글자 크기 조절: 글자 크기에 맞춰 본문 전체에 동적으로 스타일을 매핑하거나, Tailwind Typography( prose )
      플러그인의 크기 옵션( prose-sm ,  prose-base ,  prose-lg ,  prose-xl )을 동적 적용합니다.

  ──────
  ### 2. 도입 시 얻을 수 있는 장점 (UX 관점)

  1. 디바이스 대응력 향상 (고해상도 모니터)
      • 사용자의 모니터 해상도(노트북의 작은 화면부터 27인치 이상의 QHD/4K 모니터, 울트라와이드 모니터까지)에 맞춤형
      가독성을 제공할 수 있습니다.
  2. 다양한 연령층/시력의 사용자 편의성
      • 글자 크기를 키우고 싶어 하는 사용자(예: 노안이 있거나 작은 폰트가 피로한 사용자)에게 매우 유용한 접근성을
      제공합니다.
  3. 표(Table) 및 이미지 가독성 확보
      • ERP 매뉴얼 특성상 화면 캡처 이미지나 데이터 테이블이 자주 등장합니다. 본문 너비를 넓힐 수 있게 유연성을 주면
      테이블 컬럼이 잘리거나 이미지가 너무 작게 표현되는 문제를 사용자가 직접 극복할 수 있습니다.

  ──────
  ### 3. 의견 요약

  이 기능은 시스템 전체의 디자인 정체성을 훼손하지 않으면서도 사용자의 접근성과 가독성 만족도를 극도로 끌어올릴 수 있는
  훌륭한 기능 개선 아이디어입니다. 추후 코드로 구현을 원하실 때 언제든 말씀해 주시면 화면 설계서에 맞춰 깔끔하게 연동해
  드리겠습니다.

## folders에 is_use 추가

  서버 환경 및 로컬 개발 환경에서 데이터베이스 파일( .db )에 컬럼 추가 SQL 명령을 내리는 두 가지 방법을 안내해
  드립니다.

   sqlite3  명령어 뒤에 경로와 쿼리문을 연달아 넘겨주는 한 줄 명령(CLI) 방식이 가장 안전하고 빠릅니다.
  ──────
  ### 1. 로컬 개발 환경 (Local PC)

  로컬 개발 환경의 기본 경로는  /home/kdy987/work/aman/aman-base-dir/db/aman.db  입니다.
  터미널에서 아래 명령어를 한 줄로 복사하여 붙여넣으시면 즉시 반영됩니다.

    sqlite3 /home/kdy987/work/aman/aman-base-dir/db/aman.db "ALTER TABLE folders ADD COLUMN is_use INTEGER DEFAULT 1;"

    sqlite3 /data/asseterp/aman/db/aman.db "ALTER TABLE folders ADD COLUMN is_use INTEGER DEFAULT 1;"
    ──────
  ### 2. 서버 운영 환경 (jskn 서버)

   application-jskn.properties  설정 파일에 정의된 서버 데이터베이스 경로는  /data/docker/apps/aman/db/aman.db  입니다.
  서버에 SSH로 접속하신 상태에서 다음 명령어를 실행해 주시면 됩니다.

    sqlite3 /data/docker/apps/aman/db/aman.db "ALTER TABLE folders ADD COLUMN is_use INTEGER DEFAULT 1;"
    sqlite3 /data/asseterp/aman/db/aman.db "ALTER TABLE folders ADD COLUMN is_use INTEGER DEFAULT 1;"
    ──────
  ### 🔍 정상 반영 여부 검증 (선택 사항)

  컬럼이 에러 없이 잘 들어갔는지 뼈대(Schema)를 조회해보고 싶으시다면 다음 명령어로 컬럼 목록을 확인하실 수 있습니다.

    # folders 테이블의 DDL 문장 확인
    sqlite3 /data/docker/apps/aman/db/aman.db ".schema folders"

  출력 결과물 마지막 즈음에  is_use INTEGER DEFAULT 1 이 정상적으로 적혀있다면 모든 처리가 완료된 것입니다.

### 1.4.7

- Rock기능 page에 대해서 rock 기능이 필요하다.

  - 이유는 A가 1번 페이지를 편집하는데, B가 역시 그 페이지를 수정한다면 나중에 저장하는 사람의 것이 최종저장되고 만다.
  - Doc User A가 rock을 걸어 놓고 수정하면 B는 1번 페이지를 열었을 때 rock이 걸려 있음을 인지해야한다.
  - Doc User A가 rock을 걸어 놓고 있지 않은 상태에서 작업하는데. 그 페이지를 B가 rock을 걸었다면 A가 저장할 때 rock이 걸려서 저장할 수 없다고 알려준다.
  - Doc User A가 rock을 걸어 놓은 페이지는 A또는 Admin이 해제 할 수 있다.
  - Doc User A가 rock을 걸어 놓고 편집중인데, admin이 해제를 하면 A가 저장시 admin에 의해서 rock이 해제되었다고 알려준다. 저장은 된다.
  - Admin이 rock을 걸어 놓은 것은 Admin만이 해제 가능하다. 즉 Doc User는 수정한 후 저장할 수 없다.

- Rock기능을 위해서 새로운 table이 즉 page-history table이 필요할까? 아니면 page에 컬럼 rock-status, rock-user, rock-time 를 추가하는 것으로 족할까?
- Rock기능에 미처 생각치 못한 상황이 있을까?
